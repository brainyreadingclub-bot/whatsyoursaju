# 기술 스택 및 시스템 구조 — SAJU (What's Your Saju)

---

## 1. 기술 스택

### 현재 스택 (무료 기능)

| 영역 | 기술 | 선택 이유 |
|------|------|-----------|
| **언어** | JavaScript (Vanilla) | 프레임워크 없이 브라우저 네이티브 실행, 의존성 0 |
| **마크업/스타일** | HTML5 + CSS3 (인라인) | 단일 파일 배포의 극단적 단순성 |
| **폰트** | Google Fonts (Noto Serif KR, Cormorant Garamond, Nanum Myeongjo) | 동양적 서체 + 서양 세리프 조합 |
| **음력 변환** | korean-lunar-calendar v0.3.6 (CDN) | 음력→양력 변환 |
| **이미지 생성** | html2canvas (CDN) | 클라이언트에서 공유 카드 생성 |
| **배포** | Vercel (정적 호스팅) | 무료 티어, 전역 CDN, 자동 배포 |
| **분석** | Google Analytics (G-LTFZ1YMVNM) | 전환 추적, 이벤트 분석 |
| **이메일 수집** | Formspree | 사전 예약 폼 (결제 도입 전까지) |

### 유료 기능 스택 (Phase별 점진 도입)

| 컴포넌트 | Phase 1 | Phase 2+ | 선택 근거 |
|---------|---------|----------|-----------|
| **서버리스** | Vercel Functions | 동일 | 이미 Vercel 배포 중, 추가 인프라 불필요 |
| **데이터 저장** | Vercel KV (Redis) | Supabase (PostgreSQL) | 최소 인프라 원칙 — 필요해질 때만 확장 |
| **결제** | PortOne (구 아임포트) | 동일 | 한국 PG 올인원 (카카오페이/토스/카드) |
| **AI** | Claude API (Sonnet) | 동일 | 한국어 해석 품질 최상, 건당 원가 ~100원 |
| **인증** | 이메일 인증 (무료 맛보기용) | Supabase Auth (소셜 로그인) | 점진 도입 |

### 의도적으로 사용하지 않는 것

| 미사용 기술 | 이유 |
|-------------|------|
| 자체 서버 인프라 (AWS EC2 등) | Vercel 서버리스로 충분. 관리 오버헤드 제거 |
| 자체 인증 시스템 | Supabase Auth 활용. 바퀴 재발명 불필요 |
| 자체 AI 모델 | Claude API 활용. 자체 학습 ROI 불분명 |
| React/Vue/Svelte | 현재 규모에서 프레임워크 오버헤드 불필요 |
| 빌드 시스템 (Webpack/Vite) | 단일 HTML + 서버리스 함수의 하이브리드 유지 |
| 네이티브 앱 | 웹 우선 전략, PWA로 충분 |
| GraphQL | REST API로 충분한 규모 |

### Phase별 기술 스택 진화

```
[현재] index.html + Vanilla JS + Vercel CDN + Formspree
   │
[Phase 0.5] + Claude API 직접 호출 (베타 테스트용)
   │
[Phase 1] + Vercel Functions (/api/*) + PortOne + Vercel KV
   │
[Phase 2] + Supabase (PostgreSQL + Auth) + 멀티턴 세션 관리
   │         + saju-engine.js 분리 (index.html과 테스트 코드 동기화 해결)
   │
[Phase 3] + Supabase Auth (소셜 로그인) + PDF 생성 + 알림 시스템
```

```

│  │  index.html      │   │  유료 기능 JS           │  │
│  │  (무료 분석 전체) │   │  (결제/AI/그룹 호출)   │  │
│  │  변경 없음        │   │  Phase 1에서 추가      │  │
│  └──────────────────┘   └────────────────────────┘  │
└──────────────┬──────────────────────┬───────────────┘
               │                      │
       폰트/CDN만 요청          fetch(/api/*)
               │                      │

┌──────────────────┐    ┌─────────────────────────────┐
│  외부 CDN        │    │  Vercel Serverless Functions │

│  · html2canvas   │    │  /api/counsel  → Claude API │
│  · lunar-cal     │    │  /api/create-order → 주문   │
│                  │    │  /api/verify   → 결제 검증  │
│                  │    │  /api/webhook  → PG 웹훅    │
│                  │    │  /api/group    → 그룹 분석  │
│                  │    │  /api/session  → 세션 관리  │
└──────────────────┘    └──────────┬──────────────────┘
                                   │
                    ┌──────────────┼──────────────┐
                    ▼              ▼              ▼
              ┌──────────┐  ┌──────────┐  ┌──────────┐
              │ Claude   │  │ PortOne  │  │ Vercel   │
              │ API      │  │ (결제)   │  │ KV/Supa  │
              │ (AI상담) │  │          │  │ (저장소) │
              └──────────┘  └──────────┘  └──────────┘
```

### 핵심 포인트

- **무료 기능은 서버 없이 100% 클라이언트 동작** — 기존과 완전 동일, 변경 없음
- **유료 기능은 Vercel Serverless를 통한 최소 백엔드** — /api/* 경로에만 존재
- **결제/AI 장애 시 무료 서비스에 영향 없음** — 격리 원칙
- **외부 CDN 의존 = 3개**: Google Fonts, html2canvas, korean-lunar-calendar

## 3. API 설계

| 엔드포인트 | 메서드 | 역할 | Phase |
|-----------|--------|------|-------|
| `/api/counsel` | POST | AI 상담: 사주 컨텍스트 + 질문 → Claude 응답 | 1 |
| `/api/create-order` | POST | 결제 주문 생성 (서버에서 금액 확정 — 변조 방지) | 1 |
| `/api/verify-payment` | POST | PortOne imp_uid로 결제 검증 + 액세스 토큰 발급 | 1 |
| `/api/webhook` | POST | PortOne 결제 웹훅 수신 | 1 |
| `/api/group` | POST | 그룹 분석: N명 사주 데이터 + AI 해석 | 2 |
| `/api/session` | GET/POST | AI 멀티턴 세션 저장/조회 | 2 |

### 결제 플로우

```
클라이언트: 상품 ID ("ai-single") 전송
    ↓
/api/create-order: 상품 ID → 금액 조회(서버 상수) → 주문번호 생성 → 반환
    ↓
클라이언트: PortOne SDK로 결제창 오픈 (서버에서 받은 주문번호+금액 사용)
    ↓
결제 완료 → 클라이언트: imp_uid 수신
    ↓
/api/verify-payment: imp_uid로 PortOne API 조회 → 실제 금액 == 상품 금액 확인 → 토큰 발급
    ↓
클라이언트: 토큰 저장 (localStorage) → /api/counsel 호출
```

### AI 상담 플로우

```
클라이언트: { token, sajuContext, question } → /api/counsel
    ↓
서버: 토큰 검증 → 사주 컨텍스트 직렬화 → Claude API 호출
    ↓
Claude: 시스템 프롬프트(명리학 프레임워크) + 사주 데이터 + 사용자 질문
    ↓
서버: Claude 응답 → 클라이언트에 반환
```

## 4. AI 연동 아키텍처

### 사주 컨텍스트 직렬화 형식

index.html에서 계산된 `window._lastAnalysisResult`를 확장하여 AI에 전달:

```json
{
  "name": "임재윤",
  "gender": "남",
  "birthDate": "1994-10-11",
  "birthHour": 7,
  "pillars": {
    "year": { "gan": 0, "ji": 10, "hanjaGan": "甲", "hanjaJi": "戌" },
    "month": { "gan": 0, "ji": 10, "hanjaGan": "甲", "hanjaJi": "戌" },
    "day": { "gan": 6, "ji": 6, "hanjaGan": "庚", "hanjaJi": "午" },
    "hour": { "gan": 9, "ji": 7, "hanjaGan": "癸", "hanjaJi": "未" }
  },
  "elementCounts": { "목": 2, "화": 1, "토": 3, "금": 1, "수": 1 },
  "yongsinInfo": { "신강여부": "신강", "용신": "수", "희신": "목", "기신": "토" },
  "daeunList": [{ "gan": 1, "ji": 11, "age": 9 }, ...],
  "hapChung": { "합": [...], "충": [...], "형": [...], "파": [...], "해": [...] },
  "sinsalList": [{ "name": "화개살", "position": "년지" }, ...],
  "specialStructures": [{ "type": "쌍편재", "desc": "..." }],
  "yearFortune": { "등급": "보통", "천간운": "재성", "지지운": "식상" }
}
```

### 시스템 프롬프트 구조

```
[역할 정의] 전통 명리학 전문가 + 상담가
[해석 프레임워크]
  - 일간 강약 판단법
  - 용신/기신 활용 원칙
  - 합충형파해의 실생활 적용
  - 대운/세운 교차 해석법
  - 궁합 해석 원칙 (천간합/지지충의 관계 의미)
[품질 원칙] 계산 결과를 읽어주는 것이 아니라, 맥락을 읽어 처방하는 것
[금지사항] 의료/법률/투자 구체적 조언 금지, 맹신 유도 금지
[사주 데이터] {위의 JSON 컨텍스트}
```

### 토큰 사용량 및 비용 추정

| 항목 | 추정 토큰 | 비용 (Sonnet) |
|------|-----------|-------------|
| 시스템 프롬프트 + 사주 컨텍스트 | ~3,000 input | ~$0.009 |
| 사용자 질문 | ~200 input | ~$0.0006 |
| Claude 응답 | ~1,500 output | ~$0.0225 |
| **건당 합계** | | **~$0.032 (~40원)** |

일 100건 기준 월 ~$100, 일 300건 기준 월 ~$300.

### 품질 핵심

> **제품 품질의 70%가 프롬프트 엔지니어링에 달려있다.**
> 시스템 프롬프트의 명리학 해석 프레임워크가 얼마나 정교한가가
> "만족스러운 AI 상담"과 "GPT한테 사주 물어본 것"의 차이를 결정한다.

## 5. 유저 흐름 구조

### 무료 개인 분석 (기존 — 변경 없음)

```
[사용자 방문] → [이름·성별·생년월일시 입력] → [분석 시작]
  → [로딩 2.5초] → [결과 페이지] → [공유/궁합미니게임]
```

### AI 상담 (Phase 1 — 신규)

```
[결과 페이지] → [AI 맛보기 영역 노출]
  → [질문 입력] → [이메일 인증 (첫 1회 무료)]
  → [AI 응답 표시] → [더 깊은 대화 원하면 → 결제]
  → [PortOne 결제창] → [결제 완료]
  → [추가 질문 가능 (상품별 횟수)]
```

### 프리미엄 궁합 (Phase 1 — 신규)

```
[궁합 미니게임 결과] → [상세 분석 블러 영역]
  → [결제] → [AI 기반 심층 궁합 해석 표시]
```

### 그룹 분석 (Phase 2 — 신규)

```
[그룹 분석 모드 진입] → [1인씩 정보 입력 (2~5인)]
  → [결제] → [로딩] → [그룹 결과 페이지]
    ├─ 각 개인 요약
    ├─ nC2 쌍대 궁합 매트릭스
    ├─ 팀 오행 분포 + 역할 분화
    ├─ 10년 타임라인 (Phase 2)
    └─ AI 기반 실행 인사이트
```

## 6. 데이터 흐름 구조

### 무료 분석 파이프라인 (기존 — 변경 없음)

```
입력 검증 → (음력 시 양력변환) → 4주 계산 → 오행 카운트
  → 신강/신약 → 용신/조후용신 → 십신·12운성 → 합충형파해
  → 신살·공망 → 영역별 풀이 → 세운 3년 → 대운
  → 결과 렌더링 → window._lastAnalysisResult 저장
```

### AI 상담 데이터 흐름 (Phase 1 — 신규)

```
window._lastAnalysisResult (클라이언트)
  → JSON 직렬화 → /api/counsel (서버리스)
  → 시스템 프롬프트 + 사주 컨텍스트 + 질문 → Claude API
  → Claude 응답 → 클라이언트 렌더링
```

### 그룹 분석 데이터 흐름 (Phase 2 — 신규)

```
[N명의 입력] → [각 1인씩 개별 분석 (기존 파이프라인 × N)]
  → [N명의 분석 결과 배열] → /api/group (서버리스)
  → [서버: 교차 비교 계산]
      ├─ nC2 쌍대 궁합 (천간합/지지충/오행 상생극)
      ├─ 그룹 오행 합산 + 역할 분화
      └─ 대운 교차 타임라인
  → [Claude API: 관계 해석 + 팀 역학 + 실행 인사이트 생성]
  → 결과 JSON → 클라이언트 렌더링
```

### 데이터 구조 (기존 + 확장)

| 데이터 | 형태 | 비고 |
|--------|------|------|
| 기둥 (Pillar) | `{ gan: number, ji: number }` | 천간/지지 인덱스 (0-indexed) |
| 오행카운트 | `{ 목: n, 화: n, 토: n, 금: n, 수: n }` | 장간 가중치 합산 |
| 대운 | `{ gan, ji, age }[]` | 8개, 10년 단위 |
| 궁합 결과 (신규) | `{ score, grade, chemistry, personalityScore, loveScore, workScore, details }` | details에 AI 해석 포함 |
| 그룹 결과 (신규) | `{ individuals[], pairs[], teamDynamics, timeline, insights }` | MECE 구조 |

## 7. 전환 원칙 (기존 서비스 보호)

1. **index.html은 그대로 동작** — 무료 사용자 경험 변화 없음
2. **서버리스 함수는 /api/* 에만 추가** — 기존 코드와 물리적 격리
3. **결제/AI 장애 시 무료 기능에 영향 없음** — 독립적 장애 격리
4. **인프라는 필요해질 때만 추가** — 최소 인프라 원칙 (Phase 1: KV만, Phase 2: Supabase)
5. **Feature flag로 유료 기능 점진적 오픈** — 전체 사용자에게 일괄 오픈하지 않음

## 8. 보안 및 어뷰징 방지

| 위협 | 대응 |
|------|------|
| 결제 금액 변조 | 서버에서 상품 ID → 금액 조회 (클라이언트 금액 무시) |
| 무료 AI 맛보기 어뷰징 | 이메일 인증 기반 1인 1회 제한 (Vercel KV에 이메일 해시 저장) |
| API 키 노출 | Claude/PortOne API 키는 서버리스 함수 환경변수에만 저장 |
| Rate limiting | Vercel Edge Config 또는 Vercel KV 기반 IP/토큰별 제한 |
| 결제 토큰 위조 | 서버에서 발급한 JWT/UUID만 유효, Vercel KV에서 검증 |

## 9. 운영 비용 구조

### Phase 1 예상 비용 (일 100건 유료 결제 기준)

| 항목 | 무료 티어 한도 | 예상 월비용 |
|------|--------------|------------|
| Vercel Functions | 100GB-hours/월 | $0 (한도 내) |
| Vercel KV | 3,000 요청/일 무료 | $0 (한도 내) |
| Claude API (Sonnet) | 종량제 | ~$100 (~13만원) |
| PortOne PG 수수료 | 건당 2.5~3.5% | 매출의 3% |
| Google Fonts CDN | 무료 | $0 |
| **합계 (고정비)** | | **~$100/월 + PG 수수료** |

### 손익분기점

- AI 상담 건당 원가: ~40원
- AI 상담 건당 매출: 1,900~3,900원
- **건당 마진: 97%+**
- 월 고정비 $100 = 약 13만원
- **일 4~5건 유료 결제로 고정비 회수**

## 10. 현재 구조의 강점과 한계

### 강점

| 강점 | 효과 |
|------|------|
| 무료 기능 서버 비용 0원 | 트래픽 증가해도 Vercel 무료 티어로 충당 |
| 유료 기능 종량제 | 매출이 없으면 비용도 거의 없음 |
